# OpenAPI 平台 - 技术选型与架构设计

## 一、技术选型

### 1.1 后端技术栈

| 技术 | 选型 | 理由 |
|------|------|------|
| Web 框架 | **FastAPI** | 现代、高性能、自动生成 API 文档、支持异步、Python 类型提示 |
| 数据库 | **SQLite** | 开发阶段轻量易用，无需额外安装；生产可迁移到 PostgreSQL |
| ORM | **SQLAlchemy** | Python 主流 ORM，功能强大 |
| 认证库 | **PyJWT** | JWT 生成和验证 |
| 密码加密 | **bcrypt** | 安全可靠 |
| 数据验证 | **Pydantic** | FastAPI 内置，数据验证和序列化 |

### 1.2 前端技术栈

| 技术 | 选型 | 理由 |
|------|------|------|
| UI 框架 | **Vue 3 + Element Plus** | 简洁的管理后台组件 |
| 构建工具 | **Vite** | 快速开发启动 |
| HTTP 客户端 | **Axios** | 与后端 API 交互 |
| 路由 | **Vue Router** | 单页应用路由 |

### 2.2 认证流程设计

#### 2.2.1 API 密钥认证流程

```
调用方请求: GET /openapi/user/info?app_id=xxx&sign=xxx

                    ▼
            ┌───────────────┐
            │  网关接收请求  │
            └───────┬───────┘
                    ▼
        ┌─────────────────────┐
        │  验证 AppID 存在     │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │  验证 AppSecret     │
        │  (比对签名)          │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │  验证接口访问权限    │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │  转发请求到后端服务  │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │  返回响应给调用方    │
        └─────────────────────┘
```

#### 2.2.2 OAuth2.0 授权码模式流程

```
1. 用户访问: GET /oauth/authorize?client_id=xxx&redirect_uri=xxx&response_type=code
              │
              ▼
2. 显示授权页面，用户点击"授权"
              │
              ▼
3. 生成授权码 code，重定向到: {redirect_uri}?code=xxx
              │
              ▼
4. 应用服务器用 code 换取令牌:
   POST /oauth/token (grant_type=authorization_code, code=xxx, client_id=xxx)
              │
              ▼
5. 返回 access_token 和 refresh_token
              │
              ▼
6. 调用接口: GET /openapi/user/info?access_token=xxx
```

```
        │  转发请求到后端服务  │

        ┌─────────────────────┐
        │  返回响应给调用方    │
        └─────────────────────┘
```

### 2.3 核心设计原则

1. **分层清晰**: API 层 → 服务层 → 数据层，各层职责明确
2. **配置分离**: 敏感配置（密钥、数据库）通过环境变量或配置文件管理
3. **安全性优先**: 密码加密存储、API 签名验证、JWT 时效控制
4. **可扩展性**: 模块化设计，便于后续添加新功能
5. **开发友好**: FastAPI 自动生成文档，便于调试和测试
